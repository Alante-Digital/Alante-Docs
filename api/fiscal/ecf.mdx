## Descripción general

Esta API permite a sistemas externos generar comprobantes fiscales electrónicos (e-CF) que serán procesados y enviados a la **DGII** (Dirección General de Impuestos Internos) de la República Dominicana. Si tu empresa necesita integrar facturación electrónica con otro sistema, esta API está pensada para ser clara, segura y fácil de implementar.

Cuando usas el flujo **estándar** (producción, certificación o pruebas **con** persistencia en Alante), el sistema se encarga de:

- Asignar el siguiente **NCF** (Número de Comprobante Fiscal) disponible según los lotes de la organización
- Generar el **XML** según las especificaciones de la DGII
- **Firmar** digitalmente el XML con el certificado de la organización
- **Enviar** el comprobante a la DGII
- **Consultar** el estado del comprobante en la DGII
- Generar **PDF**, datos para **código QR** y acceso al **XML** cuando el comprobante es aceptado
- **Almacenar** el documento para consulta posterior (UUID, módulo Fiscal, endpoints de esta API)

Si envías **`"environment": "testecf"`** y el campo **`numero`** (eNCF de prueba), aplica el **modo pruebas sin persistencia** (ver más abajo): no se consumen lotes ni se guarda el comprobante en Alante; sirve para validar integración y respuestas de la DGII en TesteCF.

## Beneficios clave

- **Integración simple** con endpoints claros
- **Cumplimiento** con reglas DGII en el armado del XML y el envío
- **Modo TesteCF opcional** para probar la integración sin crear comprobantes en el ERP
- **Documentos listos** (PDF, XML, QR vía API) cuando el flujo es el estándar y la DGII acepta el comprobante

## Autenticación

Todas las peticiones requieren autenticación mediante **API Key** asociada a una organización.

### Obtener una API Key

1. Inicia sesión en Alante Digital
2. Ve a **Configuración** → **Integraciones**
3. Crea una nueva API Key
4. Copia la clave generada (**solo se mostrará una vez**)

### Formato de autenticación

**Opción 1: Header (recomendado)**

```
X-API-Key: alante_xxxxxxxxxxxxx
```

**Opción 2: Parámetro en URL**

```
?api_key=alante_xxxxxxxxxxxxx
```

**Opción 3: Bearer token**

```
Authorization: Bearer alante_xxxxxxxxxxxxx
```

## Base URL

```
https://alante.digital/api/external/fiscal
```

Las rutas documentadas (`/invoices`, etc.) se concatenan a esta base (por ejemplo: `POST https://alante.digital/api/external/fiscal/invoices`).

## Endpoints disponibles

### 1. Crear comprobante fiscal

Genera un nuevo comprobante fiscal electrónico y lo envía a la DGII (flujo estándar), **o** ejecuta una **prueba TesteCF sin persistencia** si indicas `environment: testecf` y `numero`.

**Endpoint:** `POST /invoices`

**Headers:**

```
Content-Type: application/json
X-API-Key: alante_xxxxxxxxxxxxx
```

**Body (JSON) — ejemplo flujo estándar (NCF asignado por Alante):**

```json
{
  "tipo_comprobante": "E31",
  "fecha_emision": "2026-04-23",
  "environment": "certecf",
  "rnc_comprador": "123456789",
  "razon_social_comprador": "Cliente XYZ SRL",
  "razon_social_emisor": "Mi Empresa SRL",
  "direccion_emisor": "Calle Principal #123",
  "correo_emisor": "contacto@miempresa.com",
  "items": [
    {
      "nombre": "Producto 1",
      "descripcion": "Descripción detallada del producto",
      "cantidad": 2,
      "precio_unitario": 100.0,
      "descuento": 0,
      "tasa_itbis": 18,
      "indicador_bien_servicio": "1"
    }
  ],
  "formas_pago": [{ "codigo": "01", "monto": 236.0 }]
}
```

**Body (JSON) — ejemplo modo pruebas TesteCF sin persistencia:**

En este modo **debes** enviar el eNCF completo en **`numero`** (el integrador elige el valor de prueba; si la DGII indica que la secuencia no se utilizó, puedes reintentar u otro NCF según las reglas de prueba).

```json
{
  "environment": "testecf",
  "numero": "E310000000099",
  "tipo_comprobante": "E31",
  "fecha_emision": "2026-04-23",
  "rnc_comprador": "123456789",
  "razon_social_comprador": "Cliente de prueba",
  "items": [
    {
      "nombre": "Producto de prueba",
      "descripcion": "Línea para TesteCF",
      "cantidad": 1,
      "precio_unitario": 100.0,
      "descuento": 0,
      "tasa_itbis": 18,
      "indicador_bien_servicio": "1"
    }
  ],
  "formas_pago": [{ "codigo": "01", "monto": 118.0 }]
}
```

#### Parámetros requeridos (todos los modos)

| Parámetro | Tipo | Descripción |
|-----------|------|-------------|
| `tipo_comprobante` | string | Valores válidos: `B01`, `B02`, `B04`, `B14`, `B15`, `E31`. Para factura electrónica de crédito fiscal use `E31`. |
| `fecha_emision` | string | Fecha de emisión en formato `YYYY-MM-DD`. |
| `rnc_comprador` | string | **Obligatorio para E31.** RNC (9 dígitos) o cédula (11 dígitos). |
| `razon_social_comprador` | string | Razón social del comprador. |
| `items` | array | Mínimo un ítem. |

#### Parámetros opcionales (flujo estándar)

| Parámetro | Tipo | Descripción |
|-----------|------|-------------|
| `environment` | string | `ecf` (producción), `certecf` (certificación), `testecf` (pruebas DGII). Si **no** se envía, se usa el ambiente asociado al certificado de la organización (salvo comportamiento del servidor en entornos locales; ver notas). |
| `razon_social_emisor` | string | Si no se envía, se usa la de la organización. |
| `direccion_emisor` | string | Obligatoria en el XML si no está en la organización. |
| `correo_emisor` | string | Correo del emisor. |
| `fecha_limite_pago` | string | `YYYY-MM-DD`. Si se omite, se usa la fecha de emisión. |
| `formas_pago` | array | Formas de pago (código DGII y monto). |

#### Parámetros del modo `testecf` (sin persistencia)

| Parámetro | Tipo | Descripción |
|-----------|------|-------------|
| `environment` | string | Debe ser exactamente **`testecf`** para activar este modo. |
| `numero` | string | **Obligatorio** en este modo: eNCF completo de **13** caracteres (`E` + 2 dígitos de tipo + 10 dígitos), por ejemplo `E310000000099`. Debe **coincidir** con `tipo_comprobante` (p. ej. `E31` → el `numero` debe empezar por `E31`). |

#### Modo `testecf` sin persistencia: qué obtiene el integrador

- Respuesta con la misma idea general (`success`, `fiscal_document`, `dgii`).
- Campo adicional **`testecf_sin_persistencia: true`**.
- **`fiscal_document.uuid`**: `null` (no hay documento guardado; los `GET /invoices/{uuid}` no aplican).
- **No** se incluyen `pdf_url`, `qr_url` ni `xml_url` (no existe documento persistente ni archivos expuestos por UUID).
- **No** se consumen secuencias de lotes fiscales en Alante.
- Los XML en disco del servidor se usan solo durante la petición y se eliminan al terminar.

Puede seguir habiendo **lectura** de ajustes de organización y certificado, y la renovación del **token DGII** puede actualizar datos del certificado en base de datos como en el resto del flujo e-CF.

#### Estructura de ítems

| Campo | Tipo | Requerido | Descripción |
|-------|------|-----------|-------------|
| `nombre` | string | Sí | Nombre del producto o servicio. |
| `descripcion` | string | Sí | Descripción detallada. |
| `cantidad` | number | Sí | Máximo 2 decimales. |
| `precio_unitario` | number | Sí | Máximo 2 decimales. |
| `descuento` | number | No | Monto del descuento. |
| `tasa_itbis` | number | Sí | `0`, `16` o `18`. |
| `unidad_medida` | string | No | Solo se incluye en el XML si tiene valor. |
| `indicador_bien_servicio` | string | Sí | `1` bienes, `2` servicios. |

#### Estructura de formas de pago

| Campo | Tipo | Requerido | Descripción |
|-------|------|-----------|-------------|
| `codigo` | string | Sí | Código DGII (`01` efectivo, `02` cheque, etc.). |
| `monto` | number | Sí | Monto pagado con esa forma. |

#### Respuesta exitosa (201) — comprobante aceptado (flujo estándar)

```json
{
  "success": true,
  "fiscal_document": {
    "uuid": "29d2c2fa-8bf1-4005-88d8-cdbc63e853eb",
    "numero": "E310000000004",
    "tipo_comprobante": "E31",
    "rnc_comprador": "123456789",
    "razon_social_comprador": "Cliente XYZ SRL",
    "monto_total": "767.00",
    "total_itbis": "117.00",
    "fecha_emision": "2026-04-23",
    "estado": "Aceptado"
  },
  "dgii": {
    "success": true,
    "track_id": "bf9f19f4-691e-487f-aa5d-3c9966164bd0",
    "estado": "Aceptado",
    "mensajes": [],
    "environment": "certecf"
  },
  "pdf_url": "https://alante.digital/api/external/fiscal/invoices/29d2c2fa-8bf1-4005-88d8-cdbc63e853eb/pdf",
  "qr_url": "https://ecf.dgii.gov.do/eCF/ConsultaTimbre?rncemisor=…",
  "xml_url": "https://alante.digital/api/external/fiscal/invoices/29d2c2fa-8bf1-4005-88d8-cdbc63e853eb/xml"
}
```

#### Respuesta exitosa / rechazo (201 o 422) — modo `testecf` sin persistencia

Ejemplo aceptado:

```json
{
  "success": true,
  "testecf_sin_persistencia": true,
  "fiscal_document": {
    "uuid": null,
    "numero": "E310000000099",
    "tipo_comprobante": "E31",
    "rnc_comprador": "123456789",
    "razon_social_comprador": "Cliente de prueba",
    "monto_total": "118.00",
    "total_itbis": "18.00",
    "fecha_emision": "2026-04-23",
    "estado": "Aceptado"
  },
  "dgii": {
    "success": true,
    "track_id": "…",
    "estado": "Aceptado",
    "mensajes": [],
    "environment": "testecf"
  }
}
```

Si la DGII rechaza con **`secuenciaUtilizada: false`**, la API puede responder `422` con mensaje orientado a reintento; en modo `testecf` suele incluirse `testecf_sin_persistencia: true` y no se crea documento en Alante.

#### Respuesta comprobante rechazado (422) — flujo estándar

Cuando la DGII rechaza el comprobante, el documento puede quedar en Alante con estado **Rechazado**, `success: false` y sin URLs de PDF/QR/XML.

```json
{
  "success": false,
  "fiscal_document": {
    "uuid": "29d2c2fa-8bf1-4005-88d8-cdbc63e853eb",
    "numero": "E310000000004",
    "estado": "Rechazado"
  },
  "dgii": {
    "success": false,
    "track_id": "bf9f19f4-691e-487f-aa5d-3c9966164bd0",
    "estado": "Rechazado",
    "mensajes": [{ "valor": "Mensaje de error de la DGII", "codigo": 1209 }],
    "environment": "certecf"
  }
}
```

#### Respuesta error de validación (422)

```json
{
  "success": false,
  "error": "VALIDATION_ERROR",
  "message": "Error de validación",
  "errors": {
    "numero": ["El campo numero es obligatorio cuando environment es testecf."]
  }
}
```

#### Respuesta error de proceso (500)

```json
{
  "success": false,
  "error": "PROCESSING_ERROR",
  "message": "Error procesando la factura: [detalle]"
}
```

### 2. Consultar comprobante fiscal

**Endpoint:** `GET /invoices/{uuid}`

Solo aplica a comprobantes creados en el **flujo estándar** (hay UUID persistido). En modo `testecf` sin persistencia no se genera UUID.

### 3. Descargar PDF

**Endpoint:** `GET /invoices/{uuid}/pdf`

Solo si el comprobante fue **aceptado** y existe documento persistente.

### 4. Obtener datos del código QR

**Endpoint:** `GET /invoices/{uuid}/qr`

### 5. Descargar XML

**Endpoint:** `GET /invoices/{uuid}/xml`

## Tipos de comprobantes válidos

| Código | Descripción |
|--------|-------------|
| `B01` | Factura de contado |
| `B02` | Factura a crédito |
| `B04` | Nota de crédito |
| `B14` | Regímenes especiales |
| `B15` | Gubernamental |
| `E31` | Factura de crédito fiscal electrónica (RNC comprador obligatorio) |

Para integración ERP típica se usa **E31**. Otros tipos electrónicos pueden ampliarse según el producto.

## Ambientes DGII

| Valor | Uso |
|-------|-----|
| `testecf` | Pruebas DGII (TesteCF). Puede usarse con persistencia en Alante (asignación de NCF por lote) **o** en modo **solo prueba** con `numero` y sin registro en Alante (ver arriba). |
| `certecf` | Certificación antes de producción. |
| `ecf` | Producción. |

URLs de referencia DGII: TesteCF y CerteCF bajo `ecf.dgii.gov.do`; producción `eCF` según especificación vigente.

## Códigos de forma de pago

| Código | Descripción |
|--------|-------------|
| `01` | Efectivo |
| `02` | Cheque |
| `03` | Transferencia bancaria |
| `04` | Tarjeta de crédito |
| `05` | Tarjeta de débito |
| `06` | Nota de crédito |
| `07` | Nota de débito |
| `08` | Letra de cambio |
| `09` | Pagaré |
| `10` | Otros |

## Generación automática de archivos (flujo estándar)

Si la DGII **acepta** el comprobante y el documento queda persistido en Alante, la respuesta del `POST` puede incluir `pdf_url`, `qr_url` y `xml_url`. Si el comprobante es **rechazado**, normalmente no habrá esas URLs.

## Estados de comprobantes

| Estado | Descripción |
|--------|-------------|
| `Aceptado` | Aceptado por la DGII |
| `Aceptado Condicional` | Aceptado con observaciones |
| `Rechazado` | Rechazado por la DGII |
| `Pendiente` / `En Proceso` | En validación |

## Códigos de error habituales

| HTTP | Error | Acción sugerida |
|------|-------|-----------------|
| 401 | `UNAUTHORIZED` | Revisar API Key (header, query o Bearer). |
| 404 | `ORGANIZATION_NOT_FOUND` / `DOCUMENT_NOT_FOUND` | Organización o UUID incorrecto. |
| 422 | `VALIDATION_ERROR` | Revisar campos y formatos (`numero` en `testecf`, RNC E31, etc.). |
| 422 | `NO_BATCH_AVAILABLE` / `NO_SEQUENCES_AVAILABLE` | Lotes o secuencias (flujo estándar); no aplica en modo `testecf` sin persistencia. |
| 422 | `CERTIFICATE_NOT_FOUND` | Configurar certificado .p12 en la organización. |
| 422 | `ORGANIZATION_CONFIG_ERROR` | RNC u otros datos fiscales incompletos. |
| 500 | `PROCESSING_ERROR` / `SIGNATURE_ERROR` | Revisar logs, certificado y contraseña. |

## Notas importantes

1. **Control del ambiente DGII:** Si envías `environment` en el JSON (`testecf`, `certecf` o `ecf`), ese valor tiene prioridad para el envío a la DGII. Si lo omites, se usa el ambiente del certificado; en servidores de **desarrollo** puede haber reglas adicionales que favorezcan TesteCF en otros módulos. Para comportamiento predecible en integración, **indica siempre `environment` explícitamente**.

2. **NCF automático:** Solo en el **flujo estándar** Alante asigna el siguiente NCF del lote. En **`testecf` + `numero`** el NCF lo define el integrador en `numero`.

3. **Fechas:** `YYYY-MM-DD`. Montos: máximo 2 decimales.

4. **UUID:** Identificador público del documento persistido; no aplica en modo `testecf` sin persistencia (`uuid` null).

5. **E31:** RNC comprador obligatorio y formato válido.

6. **Rechazos:** En flujo estándar puede crearse registro con estado Rechazado; en modo `testecf` sin persistencia **no** queda comprobante en Alante.

7. **Certificado y lotes:** Flujo estándar requiere certificado y lotes activos del tipo solicitado. Modo `testecf` sin persistencia requiere certificado para firmar; **no** requiere lotes para asignar secuencia.

8. **Campos `municipio` / `provincia`:** Pueden enviarse en la API pero no se incluyen en el XML según la implementación actual de e-CF.
